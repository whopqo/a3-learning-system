3.1 设计思想

不像传统 Web 应用在一个方法里串行调用多个函数，本系统将教学流程建模为一个有向图。
每个节点封装一个 Agent 的执行逻辑 + 一个质量门检查。
节点之间通过 pass/fail 两条边连接：
- 质量门通过 → 前进到下一个节点
- 质量门不通过 → 回到上游节点重新执行

3.2 图结构

profile ──(≥7维?)→ path ──(阶段>0?)→ resource ──(KB匹配?)→ done
  ↑ 画像不够            ↑ 无阶段             ↑ 不匹配
  └── 继续追问          └── 重新规划         └── 换知识点重试

evaluate ──(掌握≥50%?)→ done
  ↓ 没掌握
resource ←── 出针对性练习题
profile ←── 更新画像薄弱点
path ←── 调整学习路径

3.3 图引擎实现 (agents/graph.py, 363 行)

class AgentGraph:
    def __init__(self):
        # 注册5个节点, 定义流转关系
        self.nodes["profile"]  = GraphNode(profile_agent, on_pass="path", on_fail="profile")
        self.nodes["path"]     = GraphNode(path_agent, on_pass="resource", on_fail="path")
        self.nodes["resource"] = GraphNode(resource_agent, on_pass="done", on_fail="resource")
        self.nodes["tutor"]    = GraphNode(tutor_agent, on_pass="done")
        self.nodes["evaluate"] = GraphNode(evaluate_agent, on_pass="done", on_fail="resource")

    def walk(self, start_node, ctx):
        """沿图行走, 流式输出 SSE 事件给前端"""
        current = start_node
        loop_count = {}
        while current and current != "done":
            # 防死循环: 单个节点最多重试 5 次
            if loop_count.get(current, 0) >= 5:
                break

            # ① 执行当前节点 → 流式输出 progress + text
            yield from self._run_node(node, ctx)

            # ② 质量门检查
            passed, reason = self._check_node(node, ctx)
            if passed:
                current = node.on_pass    # 前进
            else:
                current = node.on_fail    # 退回上游

    特点:
    - 每个节点内部独立处理异常, 单点失败不影响其他节点
    - 资源生成节点内部用 ThreadPoolExecutor(max_workers=4) 并行生成 8 种资源
    - 评估节点失败时自动调用 _feedback_eval_to_profile() 更新画像, 再回到资源节点出题

3.4 质量门定义

| 节点     | 通过条件                  | 不通过行为        |
|----------|--------------------------|------------------|
| profile  | _asked_dims 已收集 ≥7 个维度 | 回到 profile 继续追问 |
| path     | phases 列表非空           | 回到 path 重新规划   |
| resource | 知识点在知识库中 (向量语义判断) | 换一个知识点重试     |
| evaluate | 得分 ≥ 50%               | 反馈画像 → 回到 resource 出针对性题 |

4.1 对话式学习画像构建 (F1)

目标: 通过自然语言对话收集 8 个维度的学生画像, 不用表单。

流程:
1. ProfileAgent.generate_next_question() 根据已收集维度, 用 LLM 生成一句自然的追问
2. 设置 _current_dim 记录本轮在问什么维度 (如 "ml" = ML基础)
3. 学生回答 → extract_profile_update() 根据 _current_dim 只提取当前维度的评分
4. _mark_dim_asked() 标记已问 → 防止重复
5. 8 维全满 + 6轮以上更新 → 画像完成 → 自动进入路径规划

关键技术:
- LLM 提取评分: 将学生回答 + 维度标签传给 LLM, 返回 0.15~0.9 的评分
- 关键词兜底: "没有/不会"→0.15, "一般/还行"→0.5, "不错/熟悉"→0.7, "精通"→0.9
- 安全词优先: "没问题" 先于 "没" 检查, 避免正面回答被误判为零基础
- 维度名归一化: _LONG2SHORT 映射表统一长/短/中文维度名, 避免对不上

采集的 8 维:
ml(ML基础), prog(编程水平), math(数学基础),
goal(学习目标), style(认知风格), time(时间投入),
lang(语言偏好), weak(薄弱点)


4.2 多智能体协同资源生成 (F2)

目标: 根据学生画像, 生成 8 种个性化学习资源, 所有内容基于知识库不瞎编。

生成流程:
1. 从学生消息中提取知识点 (LLM 分类 prompt 中一并返回 topic)
2. 调用 ChromaDB 检索该知识点在知识库中的相关内容 (Top-5 片段)
3. 根据画像计算个性化策略:
   - _emphasis: 零基础→概念类比, 一般→概念公式平衡, 进阶→数学推导
   - _tone: 严谨学术→正式术语+LaTeX, 生动比喻→口语化+类比
4. ThreadPoolExecutor(max_workers=4) 并行生成 8 种资源:
   - lecture_notes: Markdown 讲解文档 (根据 _emphasis + _tone 调整风格)
   - mind_map: LLM→JSON→Mermaid 思维导图 (节点名去括号防渲染崩溃)
   - exercises: 练习题 (零基础 5 题 / 一般 7 题 / 进阶 8 题, 薄弱点多加概念题)
   - reading_materials: 从知识库原文提取当前章节的阅读材料和参考文献
   - extended_reading: 从知识库提取"休息一会儿"小故事
   - code_example: Python 代码案例 (新手→详注释精简版, 熟手→完整版)
   - ppt_outline: 15-20 页教学 PPT 大纲 (可一键导出 .pptx)
   - video_script: Manim 动画代码 (简化模板保证可渲染)

防幻觉机制 (所有 prompt 均注入):
"【重要约束】你必须严格基于上方「知识库参考」的内容来生成, 不得凭空编造。
  如果知识库里没有相关内容, 请如实说「知识库暂无此内容」。"

知识库: 17 章完整机器学习教材 (西瓜书), 通过 ChromaDB 向量化存储 (180个片段),
        BAAI/bge-small-zh-v1.5 本地 Embedding (512维, CPU运行, 约100MB)


4.3 个性化学习路径规划 (F3)

目标: 不是固定的 16 章列表, 而是根据画像定制阶段式学习计划。

流程:
1. 将画像转为人读的文本摘要 (基础评分→"零基础/一般/不错", 目标/风格/薄弱点)
2. 调用 analyze_knowledge_state() 从知识图谱推断已掌握的和薄弱点
3. 从 RAG 检索知识库大纲, 从知识图谱获取章节列表
4. LLM 生成阶段式路径 JSON:
   - 阶段名称、时长、学习目标
   - 涵盖章节 (必须是知识库中存在的)
   - 补充内容、动手任务、推荐资源
5. 零基础→自动加预备知识阶段, 进阶→跳过入门章

知识图谱: NetworkX 有向图 (DiGraph)
- 线性前置: ch1→ch2→ch3→...  (从 course_data 目录自动扫描)
- 交叉依赖: 7 条边 (神经网络→线性模型, SVM→模型评估, 集成学习→决策树 等)
- get_available_nodes(): 只有前驱全部掌握的节点才能学
- get_learning_path(): 拓扑排序 + 难度排序生成学习序列


4.4 智能辅导答疑 (F4, 加分项)

目标: 多模态答疑, 不只是丢一段文字。

TutorAgent 4 级辅导策略:
- L1 直接解答: 学生明确要求时直接给答案
- L2 提示引导: 先给 1-2 个关键线索, 让学生自己思考
- L3 类比讲解: 用生活化比喻解释技术概念
- L4 苏格拉底追问: 递进问题引导学生自己推导

多模态输出:
- 文字: Markdown 格式, 含概念解释 + 示例
- 图解: Mermaid 流程图/时序图, 基于 RAG 知识库
- 微课脚本: 60 秒短视频讲解脚本 (分镜 + 旁白)
- 知识库外问题: 直接拒绝, 不编造

tutoring_level 自动选择: LLM 分析学生问题 + 画像, 智能判断最适合的辅导等级。
类型安全: int() 转换 + [1,4] 边界裁剪, 防止 LLM 返回字符串 "3" 匹配不到整数 3。


4.5 学习效果评估 (F5, 加分项)

目标: 评估 → 画像更新 → 路径调整, 形成完整闭环。

评估流程:
1. 关键词毫秒级判对错 (立即显示得分, 用户体验不卡顿)
2. EvaluateAgent LLM 异步深度分析 (错因、知识漏洞)
3. compute_mastery() 掌握度计算:
   - 参考 DeepTutor 加权算法
   - recency weights: 越新的答题权重越高
   - 置信度封顶: 答 1 次最多 0.5, 防止"蒙对"算掌握
4. 结论反哺画像: 薄弱知识点→struggling_topics, 正确知识点→mastered_topics
5. 触发路径调整: PathAgent 根据新的画像重新规划

两个评估入口:
- 弹窗 Quiz: 资源库→练习题→提交评估 (交互式选择/填空)
- 聊天评估: 对话中直接发送答案

4.6 消息分类与意图识别

目标: 一句话判断用户想要什么, 准确且快速。

三层判断:
1. 向量语义预检 (message_in_kb_scope, threshold=0.12)
   - KB 里完全不沾边的消息 → 直接拒绝 (如"今天天气怎么样")
   - 节省 LLM 调用成本
2. LLM 分类 (单次调用, 返回 category + intent + topic + resource_type + should_reject)
   - 分类: ml/casual/flow/off_topic
   - 意图: resource_generation/path_planning/tutoring/evaluation/profile_building/general_chat
   - 一次 LLM 调用完成所有判断
3. 画像阶段特殊处理:
   - 画像未完成时即使 LLM 判为 path_planning, 也不跳去生成路径
   - 先检查 is_sufficient(), 确保画像够格才放行


═══════════════════════════════════════════════════════════════
五、RAG 知识检索系统
═══════════════════════════════════════════════════════════════

架构: ChromaDB + BAAI/bge-small-zh-v1.5 + LangChain Text Splitter

五步流水线:
1. 文档摄取: 读取 course_data/ 下 17 个 .txt 文件
2. 语义分段: RecursiveCharacterTextSplitter (chunk_size=500, overlap=50)
3. 向量化: SentenceTransformer 本地编码 (normalize_embeddings=True, 512维)
4. 混合检索: ChromaDB 向量检索 (cosine 距离空间, Top-K=5)
5. 生成: 检索结果 + 学生画像 → LLM 个性化生成

关键技术:
- local_files_only=True: 模型已缓存则不联网, 离线可用
- 懒加载单例模式: _get_collection() 首次调用时初始化, 后续复用
- 重建保护: _get_collection() 每次 count() 验证存活, 死了自动刷新
- 相关性判断: check_topic_relevance() 用向量语义判断 topic 是否在 KB 中
- 阅读材料提取: extract_reading_materials() 只取当前章节, 截断到下一个 ###
- 小故事提取: extract_stories() 清理 ### 标题行和尾部序号, MD5 hash 去重

知识库规模: 17 章教材, 180 个语义片段, Embedding 模型约 100MB


═══════════════════════════════════════════════════════════════
六、技术亮点
═══════════════════════════════════════════════════════════════

1. 有向图多Agent协同引擎
   - 不是顺序调用, 而是定义了节点+边+质量门的图结构
   - 质量门不通过自动回到上游, 形成自反馈闭环
   - MAX_LOOPS=5 防止死循环

2. 防幻觉三机制
   - 输入层: 敏感词过滤 + Prompt Injection 检测
   - 生成层: 全部 LLM prompt 注入 KB_ONLY_RULE + RAG 强制溯源
   - 输出层: 知识库外内容直接拒绝, 不编造

3. 画像采集的工程化
   - 双层提取: LLM 为主 + 关键词兜底 (安全词优先防误判)
   - 维度追踪: _current_dim 记录系统在问什么, 每轮只标记一个维度
   - 长/短名归一化: 统一 _LONG2SHORT 映射表, 杜绝对不上
   - 四重防提前结束: is_sufficient + _all_asked + _update_count + round_num

4. SSE 流式输出
   - process_message_stream() Generator 逐事件 yield
   - handler 层统一做进度→text→done 三种事件转发
   - 原 Skip_routing 引起的 done 丢失 bug 已全部修复

5. 资源并行生成
   - ThreadPoolExecutor(max_workers=4) 并行调 8 次 LLM
   - 总耗时从 40-60s → 15-25s

6. 全 LLM 驱动（无硬编码关键词判断）
   - 消息分类、意图识别、维度提取、练习题识别全部走 LLM
   - 唯一保留的是极少数工具型前缀后缀清洗（如去掉"帮我生成"提取纯知识点名）

7. LLM 调用可靠性
   - 指数退避重试 (max 3 次)
   - 认证/授权错误不重试直接抛
   - 流式调用也支持重试 + token 计数
   - 支持 5 种 Provider 一键切换 (DeepSeek/讯飞星火/通义千问/智谱/Ollama)


═══════════════════════════════════════════════════════════════
七、数据流与状态管理
═══════════════════════════════════════════════════════════════

会话模型:
- 每个 session 包含: phase, profile, resources, learning_path,
  conversation_history, evaluation_history
- session 存储在 handler.sessions 字典中
- 每次对话结束 → _save_sessions() 写入 db/sessions.json
- 服务器启动 → _load_sessions() 恢复上次会话
- 2 小时不活跃自动清理

SSE 事件流:
1. 用户发送消息 → POST /api/chat
2. handler.process_message_stream() 启动 Generator
3. 每阶段 yield {"type":"progress", ...} 或 {"type":"text", ...}
4. 最终 yield {"type":"done", "result": {...}}
5. server.py 的 generate() 协程逐个转发给前端
6. 前端 sse() 函数解析事件: progress→进度条, text→逐字追加, done→更新状态

资源生成数据流:
handler._handle_resource()
  → 构建 ctx {profile, _topic, _session, _exercises_only, learning_path}
  → yield from graph.walk("resource", ctx)
  → graph._exec_resource() 检查KB → 检索ChromaDB → 并行生成8种资源
  → ctx["resources"] 写回 → sess["resources"] 同步
  → result["metadata"] = {"resources": ctx["resources"]}
  → SSE done 事件的 payload 包含 resources
  → 前端 updateState(d) → S.resources = d.resources → renderResources() 刷新

评估闭环:
submitQuiz() → POST /api/evaluate
  → 关键词快判→立即返回score
  → LLM深度分析→ai_analysis
  → wrong_kps/right_kps写入 session.profile
  → 返回 updated_profile 给前端同步

九、运行方式
═══════════════════════════════════════════════════════════════

1. 复制 .env.example 为 .env, 填入 DEEPSEEK_API_KEY=你的Key
2. 安装依赖: pip install -r requirements.txt
3. 构建知识库: python knowledge_base/build_kb.py
4. 启动: python run.py (或双击)
5. 浏览器打开 http://localhost:8000

支持切换大模型: 修改 config.py 的 LLM_PROVIDER = "spark/qwen/glm/ollama"
